# Capital Agent v1.1 — Decision Register

Companion to [capital-agent-v1.1.md](capital-agent-v1.1.md) (spec Appendix A decisions CA-001…CA-018 remain final).
This register adds the repository-grounded and founder-locked decisions. Status **Locked** = founder direction, 2026-07-12.

| ID | Decision | Status |
|---|---|---|
| CA-100 | **Scope: complete company-side Capital Agent v1.1.** Direct financier-user functionality is deferred as a *sequencing* decision, not an architectural reduction. No company-side v1.1 capability is removed to make the first delivery smaller. | Locked |
| CA-101 | **Principal-neutral, financier-compatible foundation.** `principal_type ∈ {company, financier, platform_internal}` reserved from day one; only `company` active. Foundational schemas/contracts use generic concepts (principal, mandate, task, outcome, calculation run, evidence bundle, artifact, proposal, memory profile, audit event) — never `company_*`-prefixed shapes. Future financier activation is additive (mandates, outcomes, skills, scopes, UX) and must not require replacing the shared framework or migrating company-side foundations. | Locked |
| CA-102 | **Capital Agent ↔ Finance boundary.** Finance owns canonical financial state and execution (funding requests/offers, facilities, reservations, acceptance, payments, accounts, rails, escrow, reconciliation, provider lifecycle, execution results). Capital Agent owns intelligence products (interpretation, deterministic calculations, diagnosis, scenarios, option analysis, recommendations, outcomes, calc runs, evidence, versioned artifacts, monitoring analysis, proposals). Canonical Finance objects are created/changed ONLY via: outcome → proposal → human approval → typed Finance command → **independent Finance-domain validation** (authz, principal, mandate, canonical state, proposal status, payload integrity+hash, approval, idempotency, expiry, SoD, provider requirements, eligibility) → Finance execution → canonical result. Analysis/artifact generation never auto-creates a `FundingRequest`. Finance never treats an agent recommendation as authorization. | Locked |
| CA-103 (=plan Decision A) | **No Temporal in the initial company-side implementation.** Durability/audit/replay/recovery/monitoring built on existing infrastructure: `apps/worker`, workflow-run contracts (`packages/contracts/src/workflow-runtime.ts`), replay/event mechanisms, durable-job/retry patterns. Temporal recorded as a *deferred infrastructure decision* (not a deferred product capability). No Temporal dependencies, deployment, or abstractions now. | Locked |
| CA-104 (=plan Decision B) | **No parallel agent-task persistence.** Reuse and evolve `alpha_agent_tasks` (V007) additively: add `principal_id`, `principal_type`, `mandate_id`, `outcome_id`, task-contract version, definition version. Do not duplicate lifecycle, replay logs, results, status, or audit state. | Locked |
| CA-105 (=plan Decision C) | **Hybrid Finance state referenced, not consolidated.** Relational Finance records (offer_requests, finance_offers, reservations, …) and Alpha-object Finance records (funding_request/funding_offer payloads) are referenced through typed `CanonicalObjectRef` (source layer/system, object type/table, id, organization, optional trade id, optional version, freshness/observation timestamp, access scope). The layers do not share identifiers; the Capital Agent never joins them by id and never duplicates canonical Finance state. Finance consolidation is a separate Finance-module workstream. | Locked |
| CA-106 (=plan Decision D) | **Real repository paths.** Web work targets the actual Next.js App Router under `apps/web/src/app`; the spec's illustrative `app/(workspace)/…` paths are adapted, preserving intent. | Locked |
| CA-107 (=plan Decision E) | **Canonical seven-agent taxonomy retained in contracts/architecture** (Capital, Compliance, Risk, Market & Network, Trade Operations, Audit & Monitoring, Concierge/Coordinator). Not all seven are built now. Current web marketing labels (`AGENT_CLASSES`, apps/web/src/app/intelligence/page.tsx:76–118) may remain temporarily; reconciled in Phase 7. | Locked |
| CA-108 | **First vertical slice is an early milestone, not the product.** One trade → diagnosis, trade/landed cost, transaction P&L, cash-flow timing, working-capital requirement, funding-gap, financing-options comparison, deterministic calc runs, typed evidence, assumptions, unresolved questions, versioned artifact, recommendation, optional proposal — with structured UX and inspection. Golden Cases A/B/C are its acceptance harness. After the founder feel-test: refine, then continue the full approved company-side roadmap. | Locked |
| CA-109 | **Founder feel-test checkpoint** after the first slice (product-quality gate per direction §15). It does not authorize stopping the company-side roadmap; financier-direct implementation starts only after company-side validation + explicit founder approval + a dedicated financier-scope plan. | Locked |
| CA-110 | **PR #26 is a non-authoritative, do-not-merge draft spike.** Reference-only for repository research/interaction patterns/ideas. No merging, no building from its branch, no packet-builder framing, no implicit canonical `funding_request` creation. Salvage classifications in the implementation plan §15 are design guidance for fresh implementation. | Locked |
| CA-111 | **Evidence presence is never string-matching.** No generic proof object counts as proof of a specific claim. Unknown stays unknown; missing stays missing; contradictions stay visible until resolved. (Reverses the spike's `hasEvidenceInTrade` heuristic.) | Locked |
| CA-112 | **First-class persistence where the architecture requires it** (direction §16): mandates, outcomes, calculation runs, evidence bundles/claims/references, artifacts + versions, proposals, memory candidates, operating/finance profiles, specialist requests/reads, monitoring state, eval records. "Schema discipline" is not permission to collapse materially different concepts into generic JSON. Reuse existing infrastructure only where it already represents the same concept correctly (e.g. CA-104). | Locked |
| CA-113 | **Organization-backed principal identity.** `principal_id` IS the organization uuid; `principal_type` states the role that organization plays. Database invariant: `principal_type = 'company' ⇒ principal_id = org_id` (CHECK on every principal-carrying capital table). Only company mandates may be `active` (CHECK `agent_mandates_company_only_active`) until a future migration lifts it deliberately; `financier`/`platform_internal` rows may exist as drafts but activate nothing. No separate principals table in this phase; one can be added additively when financier onboarding requires it. | Locked (hardening pass) |
| CA-114 | **Append-only audit records with a governed-purge escape.** `financial_calculation_runs`, `evidence_claims`, `evidence_references`, `evidence_bundles` (append-only from creation — documented choice), and `capital_artifact_versions` reject UPDATE and ordinary DELETE via `app.capital_append_only_guard()`. Corrections create new records (claims use `supersedes_claim_id`/contradiction links; artifacts new versions; calculations new runs). Full-organization deletion and legal-retention purges are governed operations that must run with `SET LOCAL app.allow_governed_purge = 'on'` inside a privileged transaction — without it, org-deletion cascades fail by design. | Locked (hardening pass) |
| CA-115 | **Protected-action proposals: payload freezing + approval binding + SoD at the database.** Once status leaves `draft`, all action-defining fields are frozen (trigger); a changed payload requires a new proposal (new id, hash, approval). `approved` requires `approval_request_id` + `approved_by_user_id` + `approved_at` + `approved_payload_hash` with `approved_payload_hash = payload_hash`; proposals cannot be created directly as `approved`; SoD rejects `approved_by_user_id = proposed_by_user_id` when the rule forbids it. The proposal references the existing approval domain (`approval_request_id`) — no second approval system. Principal-aware RLS (org + principal_id + principal_type) replaced the initial org-only policies on all capital tables; composite ownership FKs make cross-org/cross-principal parent references unrepresentable. | Locked (hardening pass) |
